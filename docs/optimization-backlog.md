# 待优化清单

本文件记录暂不立即实现、但后续可能影响产品体验或工程质量的优化点。

## 前端服务端数据缓存层

| 方向 | 当前做法 | 后续成熟方案 | 适合优化的触发时机 | 重点收益 | 备注 |
| --- | --- | --- | --- | --- | --- |
| Conversation messages 缓存 | `useSeoWorkspace` 内用前端内存 `Map<conversationId, messages>` 做 UI 体验缓存 | 抽成独立客户端数据层，或引入 TanStack Query / Vue Query 管理服务端数据缓存 | 阶段 3 流式输出稳定后；会话切换、刷新、后台同步逻辑继续变多时 | 切换会话更快，请求状态更清晰，缓存与刷新规则更统一 | 当前缓存只存在浏览器页面内存里，刷新后清空；真实数据源仍是后端数据库 |
| Conversation list 分页缓存 | 当前由 `useSeoWorkspace` 手动维护 `nextCursor`、`hasMoreConversations` 和列表合并 | 使用 infinite query 管理分页、去重、后台刷新和失效 | 会话列表继续增加搜索、归档、项目筛选等能力时 | 减少手写分页状态，避免重复请求和列表合并错误 | 目前列表分页逻辑还不复杂，暂不急着引库 |
| 请求去重与后台刷新 | 当前由各 hook 手动发请求和更新 ref | 统一 query key，例如 `['conversations']`、`['conversationMessages', conversationId]` | 多个组件需要同时读取同一份后端数据时 | 避免重复请求，支持 stale-while-revalidate，用户先看缓存再后台刷新 | 适合和阶段 3 的消息刷新策略一起评估 |
| 重命名 / 删除乐观更新 | 当前先请求接口，再本地更新会话列表 | mutation 乐观更新，失败后自动回滚并提示 | 会话管理操作变多，用户对即时反馈要求更高时 | 操作反馈更快，失败处理更统一 | 后端仍负责最终一致性 |
| Runtime models / balance 缓存 | 当前 `useLlmRuntime` 自己维护模型和余额请求状态 | 用 query 管理模型列表、余额、刷新状态和错误 | 顶栏、设置面板、工作台多个位置都需要 runtime 数据时 | 减少重复状态，刷新和错误提示更一致 | 这类数据属于服务端数据，不适合塞进普通 UI 状态 |
| 滚动位置记忆 | 当前用轻量前端内存保存 `conversationId -> scrollTop` | 保持独立 UI 状态层；不建议放进服务端数据缓存库 | 长会话、消息分页、虚拟列表接入时继续扩展 | 切换历史会话时恢复阅读现场，减少抖动 | 滚动位置是 UI 状态，不是后端数据；可以独立于 Vue Query |

## 当前结论

| 问题 | 结论 |
| --- | --- |
| 现在要不要立刻引 TanStack Query / Vue Query | 暂时不引。当前阶段用轻量内存缓存能解决会话切换体验，避免过早引入新依赖 |
| 后续是否值得引入 | 值得。进入流式输出、多会话并发、消息分页、后台刷新、乐观更新后，服务端数据管理会明显变复杂 |
| Pinia 和 Vue Query 的边界 | Pinia 更适合客户端 UI 状态；Vue Query 更适合后端接口数据、缓存、请求状态和失效刷新 |
| 当前最重要的架构边界 | 数据库是真实数据源；前端内存缓存只服务切换体验，不作为业务数据源 |
